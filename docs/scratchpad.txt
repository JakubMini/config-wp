    - DO: name, id, fault state (hold/off/on)
    - TC: name, id, tc type (J/K/T...), units (C/F), cjc on/off

    - channel counts from hw diagram: 16 DI, 16 DO, 4 TC, 8 AI, 4 AO, 4 PCNT, 4 PWM
    - fix in config_limits.h, single source of truth

b - human readable config file format - get it in, parse it
    - json wrapper for human, binary TLV for flash. cant be both
    - json only at upload time, not in hot path
    - cjson dep - single header + single c, MIT, ~10kb rom on m4
    - import calls SAME validating setters as the c api. no duplicate validation
    - shape: { "format":1, "system":{...}, "di":[...], "do":[...], ... }
    - need error report struct on import so operator knows which record bombed
    - export buf param-provided not static, cleaner

DONE!
c - safe storage, conflict free
    - A/B dual slot. power loss mid-write only kills one
    - per-slot header: magic u32 | format_ver u16 | rsvd u16 | seq u32 | length u32 | crc32 u32
    - crc covers payload only
    - highest seq wins, fall back to other slot if active corrupt
    - both gone -> factory defaults from rom (in mcu flash, not eeprom)
    - storage.c is memcpy stub now, real hw = spi eeprom (M95M02 or similar)
    - crc32 iso-hdlc (zlib one), ~1kb table, runtime built at init

d - architercture - non-blocking, threadsafe
    - one mutex around the in-ram cache, all getters + setters take it
    - hot readers (io thread) hit ram, never block on storage
    - writes mark dirty, eeprom mgr thread drains to flash (matches fw diagram)
    - NEVER hold mutex across a storage write (= spi io = ms+). copy out, release, write
    - watch priority inversion: eeprom mgr low prio in diagram, if io thread takes
      mutex mid-write... need to look at it
    - concurrency test: N readers + 1 writer in posix tasks, 10s run, no tears/deadlocks

e - CiA 401 stuff
    - covers most IO config natively
    - PCNT + PWM need extension into manufacturer-specific space (0x2000+ ish)
    - need to pick our own object indices for those
    - system config maps to DCF concept in canopen
    - LSS slave - maybe, not strictly needed for the exercise
    - heartbeat producer + emcy producer required for cia 301 compliance

f - forward compat (easy to get wrong)
    - old fw + new config -> skip unknown tags
    - new fw + old config -> apply defaults for new fields
    - rewrite-preserves-unknowns: if old fw rewrites a config with unknown fields,
      those fields MUST survive. lots of impls drop them silently. dont do that.
    - both directions need explicit tests

g - piecewise vs monolithic updates
    - tlv records makes this fall out for free
    - piecewise = rewrite one tag, monolithic = rewrite whole blob
    - both reduce to same encoder
    - tag = (domain<<12) | (type<<8) | (index)

h - validation / error detection
    - per setter: range check every field
    - on load: magic + format_ver + crc + length sanity
    - enum vals cross-checked against known set
    - name len bounded, null term enforced
    - error codes explicit, not just OK/FAIL

i - factory defaults
    - static const table per type, in firmware
    - used on: fresh eeprom, reset_defaults() call, new fields after fw upgrade
    - 0xFF erased eeprom -> magic mismatch -> default load. explicitly test
    - consider read-only defaults partition in mcu flash so totally hosed eeprom
      still recovers

j - dual/triple function IOs
    - readme: "some I/Os may be dual/triple function"
    - hw can do it (pin = DI or DO or PCNT depending on mux)
    - approach 1: store all records, runtime picks based on "active function" field
    - approach 2: single mux config "pin X -> function Y"
    - lean toward 2, dont waste ram on inactive records
    - need to think more about it

k - portability / hal abstraction
    - stm32g first, but must port to other vendor
    - no OS abstraction, freertos stays
    - hal layer wraps spi/i2c/gpio/tim/adc/dac
    - storage interface already abstracted (storage.h) - good
    - device drivers (eeprom, pca9555, max31856, rtc) should NOT depend on
      stm32 hal directly, only on our hal wrapper

n - architercture critique (for writeup)
